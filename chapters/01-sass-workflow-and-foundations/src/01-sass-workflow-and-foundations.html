Modern Sass Workflow

Vite compiles Sass, and the output lands in dist/. DevTools can still point back to your .scss files via source maps.

Sass is compiled, not served

This page is styled with Sass, but the browser never sees Sass directly. Vite compiles everything to plain CSS before the page loads.

  • One entry file (main.scss) defines the styling boundary.
  • Partials are composed with @use, not global imports.
  • The output is ordinary CSS — that’s what DevTools shows.

Mental model

Sass is a preprocessor. It runs before the browser ever sees your styles. Its job is to turn .scss into plain CSS.

The browser does not understand Sass. It only ever parses the compiled CSS output. By the time the page loads, Sass is already gone.

During development, Vite may inject compiled CSS into the page using an in-memory <style> tag to support fast hot updates. That is why DevTools can sometimes show <style> as the source.

Source maps bridge that gap. They allow DevTools to map a compiled CSS rule back to the exact Sass file and line that produced it.

What this means in practice

• Write Sass for structure, reuse, and safety
• Inspect CSS in DevTools
• Jump back to Sass via source maps
• Fix the problem at the source

Sass shapes the CSS you ship, but it is never part of what the browser runs.

What to inspect

Use this component to trace compiled CSS back to its Sass source.

  1. Open DevTools and select the card component below.
  2. In the Styles pane, find a rule for .wf-card or .wf-card__heading.
  3. Click the rule’s source link to jump to the originating .scss partial.
  4. Notice that Sass variables (like $space-6) do not appear in the compiled CSS — they are resolved at build time.

Expected outcome

• DevTools shows plain CSS
• Source links jump to .scss files
• Variables are resolved, not preserved

If DevTools shows <style> in dev, that’s normal with Vite. Clicking the rule should still land on the correct Sass file via source maps.

What to do

What you should observe

• Rules appear as compiled CSS
• Source links open a .scss file
• Component rules live in partials/_components.scss
• Tokens resolve back to partials/_tokens.scss

If something looks off

  • Make sure you’re running npm run dev.
  • Confirm you’re not using Live Server.
  • Hard refresh after editing Sass files.

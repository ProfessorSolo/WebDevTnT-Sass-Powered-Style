Sass + Tailwind

Sass and Tailwind Unite

Use Tailwind for layout/spacing/utility speed, and Sass for authored components, tokens, and “design logic”.

Why combine them?

  • Tailwind excels at layout, spacing, and quick composition in HTML.
  • Sass excels at authored components: tokens, mixins, functions, and reusable patterns.
  • Keep Tailwind usage obvious in markup; keep Sass usage obvious in partials.

Rule of thumb: if a style is chosen at runtime or is about layout and spacing, use Tailwind. If a style is chosen at compile time or represents a component’s internal logic, use Sass.

This demo uses Tailwind utilities + a small Tailwind component layer, and Sass for a “Neon Panel” component with variants.

Pattern

This demo splits responsibility cleanly. Tailwind handles structure and layout. Sass handles components and design logic. They do not overlap.

  • Tailwind utilities: layout grid, spacing, and typography applied directly in HTML.
  • Tailwind component layer: a small set of @apply-based helpers (chips and buttons) to reduce repetition.
  • Sass components: authored UI (.neon-panel) powered by mixins, functions, and tone variants.

Tailwind answers “how is this laid out?” Sass answers “what is this component?”

Files: tailwind.css (Tailwind) and partials/_neon-panel.scss (Sass).

What to inspect

  1. Inspect the grid wrapper: it’s Tailwind-driven layout.
  2. Inspect .neon-panel: it’s Sass-authored component styling.
  3. Open tailwind.css to see @layer components + @apply (chips/buttons).
  4. Open partials/_neon-panel.scss to see mixins/functions/conditionals powering variants.
  5. In DevTools → Sources, compare tailwind.css output vs main.scss output (two separate inputs, one page).

Goal: Tailwind for structure, Sass for authored components. Don’t make one pretend to be the other.

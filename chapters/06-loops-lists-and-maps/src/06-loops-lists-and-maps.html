Loops, Lists, and Maps

Generate families of utilities and variants from data structures. One source of truth, many classes.

Data-driven CSS generation

Sass lets us treat CSS generation as a data problem. Instead of hand-authoring dozens of selectors, we iterate over lists and maps to emit predictable class families from a single source of truth.

  • Lists let us repeat a pattern across a known set of values.
  • Maps let us generate variants from key/value data.
  • Loops let us scale a rule family without copy/paste.

The goal is not “more CSS.” The goal is fewer places to maintain the rules.

List → utility family

$gaps: 2, 4, 6;

@each $gap in $gaps {
  .u-gap-#{$gap} {
    gap: space($gap);
  }
}

Map → variant family

$tones: (
  cool: var(--cool),
  hot: var(--hot),
  ghost: transparent
);

@each $name, $color in $tones {
  .pill--#{$name} {
    background: $color;
  }
}

Lists + @each

We generate a family of spacing utilities from a single list. Each value becomes a class like .u-gap-2, .u-gap-4, .u-gap-6.

Generated class: .u-gap-2

gap is coming from .u-gap-2
tighter spacing
same markup, different class

Generated class: .u-gap-6

gap is coming from .u-gap-6
roomier spacing
still zero copy/paste CSS

Sass (list loop)

$gaps: 2, 4, 6;

@each $g in $gaps {
  .u-gap-#{$g} {
    gap: $g * 0.25rem;
  }
}

What gets emitted

.u-gap-2 { gap: 0.5rem; }
.u-gap-4 { gap: 1rem; }
.u-gap-6 { gap: 1.5rem; }

See partials/_utilities.scss (list loop). The markup stays the same; the class picks the spacing.

Maps + @each

Maps are key/value data. Each key becomes a predictable class name, and each value becomes the styling payload. Perfect for variants like tones, intents, and themes.

Same component, different map keys

pill--cool pill--hot pill--ghost

These classes are emitted by iterating the map: cool, hot, ghost.

What changes vs what stays

  • Stays: shared pill base styles (.pill).
  • Changes: tone-specific values from the map (.pill--*).
  • Result: one source of truth, many variants.

Sass (map loop)

$tones: (
  cool: (bg: rgba(120, 160, 255, 0.18), stroke: rgba(120, 160, 255, 0.45)),
  hot: (bg: rgba(255, 140, 0, 0.18), stroke: rgba(255, 140, 0, 0.45)),
  ghost: (bg: transparent, stroke: rgba(233, 238, 252, 0.35))
);

@each $name, $vals in $tones {
  .pill--#{$name} {
    background: map-get($vals, bg);
    border-color: map-get($vals, stroke);
  }
}

Output shape

.pill--cool { ... }
.pill--hot  { ... }
.pill--ghost{ ... }

See partials/_pills.scss (map loop). Inspect a pill and trace its rules back to the map entry.

@for

@for is ideal for generating numbered scales. Here we generate a type scale: .scale-1 through .scale-6. Higher numbers produce larger text.

scale-1
scale-2
scale-3
scale-4
scale-5
scale-6

Sass (@for loop)

@for $i from 1 through 6 {
  .scale-#{$i} {
    font-size: 0.75rem + ($i * 0.35rem);
  }
}

What to notice

scale-1 → smallest
scale-6 → largest
One loop → six classes

See partials/_scale.scss (@for loop). Inspect any .scale-* class and trace it back to the loop that generated it.

What to inspect

  1. In DevTools, search the compiled CSS for .u-gap-, .pill--, and .scale-.
  2. Jump back to Sass and locate the loop that generated each family.
  3. Change one list or map value and watch the entire class family update.
  4. Notice: the loop itself never appears in the output — only the generated selectors do.
